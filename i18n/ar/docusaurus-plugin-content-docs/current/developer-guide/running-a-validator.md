# تشغيل مُدقِّق

يشرح هذا الدليل كيفية إنشاء مُدقِّق (Validator) على شبكة QoreChain، وفهم نظام تصنيف المجمّعات (Pools)، وتسجيل مفتاح PQC للأمان المقاوم للحوسبة الكمومية، ومراقبة عقدتك.

:::note
يستهدف هذا الدليل الشبكة الرئيسية **`qorechain-vladi`** (معرّف سلسلة EVM هو **9801**)، العاملة منذ 7 يونيو 2026 بإصدار السلسلة **v3.1.82**. يُنصح باستخدام شبكة الاختبار **`qorechain-diana`** (معرّف سلسلة EVM هو **9800**) للتدرّب على إعدادك قبل الانطلاق الفعلي. استبدل قيمة `--chain-id` المناسبة للشبكة المستهدفة.
:::

---

## المتطلبات المسبقة

* عقدة `qorechaind` مُزامَنة بالكامل (راجع [الاتصال بشبكة الاختبار](/getting-started/connecting-to-testnet))
* حساب مموَّل برصيد لا يقل عن **1,000 QOR** ‏(1,000,000,000 uqor) للتفويض الذاتي الأوّلي
* الإلمام بنموذج [التحصيص والتفويض](/user-guide/staking-and-delegation)

## إنشاء مُدقِّق

```bash
qorechaind tx staking create-validator \
  --amount 1000000000uqor \
  --pubkey $(qorechaind comet show-validator) \
  --moniker "my-validator" \
  --commission-rate 0.10 \
  --commission-max-rate 0.20 \
  --commission-max-change-rate 0.01 \
  --min-self-delegation 1 \
  --from mykey \
  --gas auto \
  --gas-adjustment 1.3 \
  -y
```

| المعامل                        | الوصف                                              |
| ------------------------------ | -------------------------------------------------- |
| `--amount`                     | مقدار التفويض الذاتي (الحد الأدنى للحصة)           |
| `--pubkey`                     | المفتاح العام لإجماع المُدقِّق (ed25519)           |
| `--moniker`                    | اسم مقروء بشريًا لمُدقِّقك                          |
| `--commission-rate`            | معدل العمولة الأوّلي (مثلًا 0.10 = 10%)            |
| `--commission-max-rate`        | الحد الأقصى لمعدل العمولة (غير قابل للتغيير بعد الإنشاء) |
| `--commission-max-change-rate` | الحد الأقصى لمعدل تغيير العمولة اليومي             |
| `--min-self-delegation`        | الحد الأدنى من الرموز التي يجب على المشغّل تفويضها ذاتيًا |

بعد تأكيد المعاملة، تحقّق من مُدقِّقك:

```bash
qorechaind query staking validator $(qorechaind keys show mykey --bech val -a)
```

## تصنيف المجمّعات

تستخدم QoreChain **نظام تصنيف ثلاثي المجمّعات** تديره وحدة `x/qca` ‏(Quantum Consensus Allocation). في كل **1,000 كتلة**، يُعاد تصنيف المُدقِّقين في واحد من ثلاثة مجمّعات بناءً على سمعتهم وحصتهم:

| المجمّع                              | المعايير                                          | تخصيص الكتل      |
| ------------------------------------ | ------------------------------------------------- | ---------------- |
| **RPoS** ‏(Reputation Proof-of-Stake) | السمعة >= المئين السبعين والحصة >= الوسيط         | 40% من الكتل     |
| **DPoS** ‏(Delegated Proof-of-Stake)  | إجمالي التفويض >= 10,000 QOR                      | 35% من الكتل     |
| **PoS** ‏(Proof-of-Stake)             | جميع المُدقِّقين النشطين المتبقّين                 | 25% من الكتل     |

داخل كل مجمّع، يُختار مقترحو الكتل باستخدام **اختيار عشوائي مرجّح** يتناسب مع حصتهم الفعّالة. يضمن هذا التصنيف حصول المُدقِّقين ذوي السمعة العالية وذوي التفويض العالي على تمثيل عادل، مع السماح في الوقت نفسه للمُدقِّقين الأصغر بالمشاركة.

### الاستعلام عن تصنيف مجمّعك

```bash
qorechaind query qca pool-classification $(qorechaind keys show mykey --bech val -a)
```

عبر JSON-RPC:

```bash
curl -X POST http://localhost:8545 \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc": "2.0",
    "method": "qor_getPoolClassification",
    "params": ["qorvaloper1..."],
    "id": 1
  }'
```

## منحنى الربط

تُحدَّد مكافأة التحصيص للمُدقِّق بواسطة منحنى ربط (Bonding Curve) يجمع عدة عوامل:

```
R = beta * S * (1 + alpha * log(1 + L)) * Q(r) * P(t)
```

| المتغير  | الوصف                                                      |
| -------- | ---------------------------------------------------------- |
| `R`      | مقدار المكافأة                                             |
| `beta`   | معدل المكافأة الأساسي                                      |
| `S`      | الحصة الفعّالة                                             |
| `alpha`  | ثابت قياس الولاء                                           |
| `L`      | مدة الولاء (زمن التحصيص المتواصل)                          |
| `Q(r)`   | عامل جودة السمعة، ضمن النطاق \[0.75 - 1.25]                |
| `P(t)`   | مُضاعِف مرحلة البروتوكول (يتكيّف عبر دورة حياة الشبكة)     |

**أبرز النقاط:**

* **مكافأة مدة الولاء:** يحصل المُدقِّقون الذين يحصّصون بشكل متواصل على مكافآت متزايدة عبر حدّ الولاء اللوغاريتمي. وهذا يحفّز الالتزام طويل الأمد.
* **عامل جودة السمعة:** يتراوح من 0.75 (سمعة ضعيفة) إلى 1.25 (سمعة ممتازة). تُحسب السمعة من زمن التشغيل، والمقترحات الناجحة، والمشاركة المجتمعية، وجودة التحقق من المعاملات.
* **مُضاعِف مرحلة البروتوكول:** يتكيّف مع نضج الشبكة عبر مراحلها المختلفة (الإقلاع، النمو، النضج).

## القطع التصاعدي

تستخدم QoreChain نموذج **قطع تصاعدي** (Progressive Slashing) يُصعِّد العقوبات على المخالفين المتكررين مع السماح للمُدقِّقين بالتعافي بمرور الوقت:

```
penalty = base_rate * escalation^effective_count * severity
```

| المعامل                          | القيمة         |
| -------------------------------- | -------------- |
| الحد الأقصى للعقوبة لكل حدث      | 33% من الحصة   |
| نصف عمر التلاشي                  | 100,000 كتلة   |
| شدة التوقف عن العمل              | 1.0            |
| شدة التوقيع المزدوج              | 2.0            |
| شدة هجوم العميل الخفيف           | 3.0            |

1. **كل مخالفة تزيد العدّاد الفعّال.** كل مخالفة (توقف عن العمل، توقيع مزدوج، إلخ) تزيد العدّاد الفعّال للمُدقِّق، مما يؤثر على العقوبات المستقبلية.

2. **العقوبة تتصاعد أُسّيًا.** تتصاعد العقوبة بناءً على العدّاد الفعّال باستخدام الصيغة أعلاه، لذا يواجه المخالفون المتكررون عقوبات أكبر بكثير.

3. **العدّاد الفعّال يتلاشى بمرور الوقت.** يتلاشى العدّاد الفعّال بنصف عمر قدره 100,000 كتلة (\~7 أيام عند كتل مدتها 6 ثوانٍ)، مما يسمح للمُدقِّقين بالتعافي بعد فترة من السلوك الجيد.

4. **الأحداث المفردة مقابل المخالفات المتكررة.** يؤدي حدث توقف عرضي واحد إلى عقوبة طفيفة، بينما تُطلق المخالفات المتكررة عواقب متزايدة أُسّيًا.

## تسجيل مفتاح PQC

يمكن للمُدقِّقين اختياريًا تسجيل **مفتاح عام تشفيري لما بعد الكم (PQC)** باستخدام خوارزمية ML-DSA-87. يوفّر هذا أمانًا مقاومًا للحوسبة الكمومية لهوية المُدقِّق ويمكن استخدامه للتوقيع الهجين.

```bash
qorechaind tx pqc register-key <pubkey-hex> hybrid \
  --from mykey \
  --gas auto \
  -y
```

| المعامل        | الوصف                                             |
| -------------- | ------------------------------------------------- |
| `<pubkey-hex>` | مفتاح ML-DSA-87 عام بحجم 2592 بايت بترميز سداسي عشري |
| `hybrid`       | وضع التسجيل (hybrid = كلاسيكي + PQC معًا)         |

تحقّق من التسجيل:

```bash
qorechaind query pqc key <account-address>
```

:::tip
**توصية:** تسجيل مفتاح PQC اختياري لكنه موصى به بشدة للمُدقِّقين العاملين على الشبكة الرئيسية. فهو يوفّر دفاعًا استباقيًا ضد تهديدات الحوسبة الكمومية.
:::

## المراقبة

### مقاييس Prometheus

تعرض QoreChain مقاييس Prometheus على المنفذ **26660**:

```
http://localhost:26660/metrics
```

أهم المقاييس التي يجب مراقبتها:

| المقياس                         | الوصف                                           |
| ------------------------------- | ----------------------------------------------- |
| `qorechain_missed_blocks_total` | إجمالي الكتل التي فوّتها مُدقِّقك               |
| `qorechain_validator_uptime`    | نسبة زمن التشغيل عبر آخر N كتلة                 |
| `qorechain_reputation_score`    | درجة السمعة الحالية                             |
| `qorechain_pool_classification` | تعيين المجمّع الحالي (0=PoS، 1=DPoS، 2=RPoS)   |
| `qorechain_consecutive_signed`  | الكتل الموقَّعة على التوالي                     |
| `consensus_height`              | ارتفاع الكتلة الحالي                            |
| `consensus_rounds`              | جولات الإجماع للارتفاع الحالي                   |

### الاستعلام عن درجة السمعة

```bash
qorechaind query reputation score $(qorechaind keys show mykey --bech val -a)
```

عبر JSON-RPC:

```bash
curl -X POST http://localhost:8545 \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc": "2.0",
    "method": "qor_getReputationScore",
    "params": ["qorvaloper1..."],
    "id": 1
  }'
```

### فحوصات السلامة

```bash
# Node status
qorechaind status | jq '.sync_info'

# Validator signing info (uptime, missed blocks)
qorechaind query slashing signing-info $(qorechaind comet show-validator)

# Check if your validator is in the active set
qorechaind query staking validators --status bonded | grep "my-validator"
```

## أفضل الممارسات التشغيلية

1. **استخدم بنية عُقد الحراسة (Sentry Nodes).** شغّل مُدقِّقك خلف عُقد حراسة لحمايته من هجمات حجب الخدمة الموزَّعة (DDoS). لا تعرّض للشبكة العامة سوى عُقد الحراسة.

2. **أعِدّ نظام تنبيهات.** هيّئ تنبيهات للكتل الفائتة، وانخفاض زمن التشغيل، وإعادات التشغيل غير المتوقعة. تفويت بضع كتل أمر طبيعي؛ أما التفويت المستمر فسيؤدي إلى القطع.

3. **حافظ على زمن تشغيل مرتفع.** يكافئ نظام السمعة زمن التشغيل المتّسق. يؤدي التوقف الممتد إلى تدهور عامل جودة سمعتك، مما يقلّل المكافآت.

4. **حدّث البرمجيات باستمرار.** تابع إصدارات QoreChain وطبّق التحديثات فورًا. نسّق مع مجتمع المُدقِّقين لترقيات السلسلة.

5. **أمِّن مفاتيحك.** استخدم وحدة أمان عتادية (HSM) أو موقِّعًا بعيدًا لمفتاح إجماع المُدقِّق. لا تخزّن المفاتيح أبدًا على نفس الجهاز الذي تعمل عليه العقدة.

6. **سجّل مفتاح PQC.** حصّن مُدقِّقك مستقبلًا ضد التهديدات الكمومية عبر تسجيل مفتاح ML-DSA-87.

7. **راقب مجمّعك.** تتبّع تصنيف مجمّعك كل 1,000 كتلة. يمكن لتحسين سمعتك أن ينقلك من PoS إلى RPoS، مما يزيد بشكل كبير من فرص اقتراح الكتل.

## مرجع أوامر المُدقِّق

```bash
# Edit validator metadata
qorechaind tx staking edit-validator \
  --moniker "new-name" \
  --website "https://myvalidator.com" \
  --details "Description of my validator" \
  --from mykey -y

# Unjail after downtime slashing
qorechaind tx slashing unjail --from mykey -y

# Delegate additional stake
qorechaind tx staking delegate $(qorechaind keys show mykey --bech val -a) \
  500000000uqor --from mykey -y

# Withdraw rewards
qorechaind tx distribution withdraw-rewards $(qorechaind keys show mykey --bech val -a) \
  --commission --from mykey -y
```

## التحقّق في الشبكات المتصلة {#connected-networks}

اعتبارًا من إصدار السلسلة **v3.1.80**، يمكن لمُدقِّق QoreChain أيضًا المساعدة في التحقّق للشبكات المتصلة عبر [الجسر](/architecture/bridge-architecture). هذه الميزة **مقيّدة بترخيص واختيارية**:

1. **امتلاك الترخيص.** يجب أن يمتلك المُدقِّق ترخيصًا نشطًا من نوع `validator_<chain>` (أو `qcb_bridge`) للشبكة المستهدفة. يرفض المُنسِّق (Orchestrator) تشغيل عميل خارجي بدونه (إغلاق آمن عند الفشل).

2. **التفعيل يوفّر العميل تلقائيًا.** عند تفعيل الترخيص، توفّر QoreChain عميل الشبكة المطابقة على عقدتك — بتنزيل العميل المثبَّت إصداره، وتوليد إعداداته، وتشغيله تحت تنسيق QoreChain. لا يُجلب أي شيء قبل التفعيل.

3. **توفير مفاتيح الشبكة وحصتها.** مفاتيح التوقيع وحصة/مُدقِّق الشبكة الخارجية **يوفّرها المشغّل** لكل شبكة؛ تقدّم QoreChain إطار عمل المشغّلات (Drivers) وبوابة الترخيص المفروضة، وليس حصتك في السلسلة الخارجية.

توجد مشغّلات لجميع **شبكات الجسر السبع والثلاثين (37)**، مصنَّفة بحسب كيفية مشاركة المُدقِّق:

| الفئة | المشاركة | أمثلة |
| ----- | ------------- | -------- |
| مُدقِّق بلا إذن مسبق | تحصيص وتشغيل | Solana، Ethereum، Avalanche، Sui، Aptos، Cardano، Tezos، Algorand، Starknet |
| بسقف / بانتخاب / بقبول | تحصيص خاضع لسقف أو انتخاب | BSC، Polygon، Polkadot، TRON، Sei، Injective، NEAR، Hedera |
| عقدة كاملة من الطبقة الثانية (L2) | تشغيل عقدة كاملة (بدون تحصيص) | Optimism، Base، zkSync Era، Linea، Scroll، Arbitrum |
| بدون تحصيص / قائمة ثقة | مراقبة / مشاركة بدون تحصيص | Bitcoin، Filecoin، XRPL، Stellar |

:::note
تثبيت إصدارات العملاء يتم على أساس أفضل جهد؛ تحقّق من إصدار العميل الأصلي (Upstream) للشبكة المستهدفة قبل أي تفعيل في بيئة الإنتاج.
:::
